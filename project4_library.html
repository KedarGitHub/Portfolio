Library Management System

  • Get Started

Introduction

This project focuses on developing a comprehensive library management system that aims to streamline operations and eliminate manual efforts associated with managing physical inventory. The primary goal is to collect and store data related to books, library visitors, new book additions, customer rewards, and overall library management, all implemented using SQL. By creating a robust database, the project addresses challenges such as handling book transactions, collecting fines, and enhancing the efficiency of the library's operations.

The schema and tables used in the database have been meticulously defined to support various functionalities. The advantages of utilizing a database over traditional Excel files are noteworthy. Databases can handle large volumes of data, offering storage capabilities up to 1 GB per field and 32 TB per relation, whereas Excel tends to crash beyond 1 million rows. Furthermore, databases allow concurrent access by multiple users and enable security features such as row-level or column-level access control. This ensures data privacy and restricts sensitive information modification to authorized personnel only.

The target users for this project include library staff, local residents, and book procurement shops. For instance, considering the case of the UB Library, which comprises multiple branches, the current system lacks cross-functionality. Users must physically visit specific branches to search for books, leading to inefficiencies. With this project, users can seamlessly search for books across all library locations, eliminating the need for physical presence. This user-friendly approach saves time for both students and staff, promoting a hassle-free experience.

Foreign Key Deletion Scenarios - Possibility and Removal

1. The association between the 'book_author_relation' table and the 'author' table, as well as the 'book' table, signifies that the primary keys ('author_id' in the 'author' table and the primary key of the 'book' table) can only be removed after eliminating this association table initially.

2. The relationship established by the 'book_category_relation' table between the 'category' table and the 'book' table implies that the primary keys ('category_id' in the 'category' table and the primary key of the 'book' table) can only be removed after dropping this association table.

3. In the 'checkout' table, the foreign keys 'unique_book_id' and 'customer_id_card_number' are utilized. Consequently, to delete the primary keys ('customer_id_card_number' in the 'customer' table and the primary key of the 'book_copy' table, which is 'unique_book_id'), it is necessary to first drop the 'checkout' table. Only then can the primary keys of the referenced tables be removed.

4. The 'book' table employs 'book_type_id' as a foreign key referencing the 'book_type' table. To eliminate the primary key ('book_type_id') of the 'book_type' table, it is imperative to drop the 'book' table first.

5. Similarly, the 'book_copy' table uses 'book_id' as a foreign key referencing the 'book' table. Consequently, to remove the primary key ('book_id') of the 'book' table, the 'book_copy' table must be dropped beforehand.

Normalization

Normalization is a fundamental database design method that focuses on reducing data redundancy and eliminating problems like Insertion, Update, and Deletion Anomalies. It achieves this by breaking down larger tables into smaller ones and creating relationships between them. The objective of normalization in SQL is to eradicate duplicated data and ensure the logical storage of information. Following normalization rules enables more efficient organization of databases and ensures data integrity.

Boyce Codd Normal Form

BCNF and 3NF stand as vital normalization techniques in database design, serving to eradicate data redundancy and enhance data integrity. BCNF relies on functional dependencies, which denote relationships among attributes within a relation. Its objective is to eliminate non-trivial functional dependencies by ensuring each determinant is a candidate key. BCNF preserves all functional dependencies and decomposes relations into separate tables based on determinants. On the other hand, 3NF concentrates on functional dependencies, particularly targeting transitive dependencies. Its aim is to remove instances where an attribute relies on another attribute through a third one, thereby reducing data redundancy by breaking down relations into smaller tables.

All tables adhere to BCNF because the primary key uniquely determines all other attributes. Regarding 3NF, the tables are already in this form since there are no transitive dependencies.

Query Optimization

Query optimization in SQL is an ongoing process aimed at enhancing query performance by reducing execution time, minimizing disk accesses, and optimizing various cost-related factors. Efficient data access is vital for applications as it directly impacts user experience. The goal is to retrieve data swiftly, ensuring a seamless and responsive application experience.

The necessity for query optimization arises due to several factors:

1. Improved Performance: Optimization of the execution plan enhances the performance of database queries, resulting in faster response times and overall system efficiency.

2. Efficient Resource Usage: Optimized queries consume fewer system resources such as CPU, memory, and disk I/O. This efficient resource utilization allows the system to handle more concurrent queries without performance bottlenecks.

3. Scalability: Query optimization becomes crucial as data volume and user access increase. Optimized queries ensure effective system scaling and enable handling larger workloads without significant performance degradation.

4. Cost Savings: Query optimization reduces the cost of executing queries by minimizing disk I/O operations and optimizing resource usage. This leads to savings in hardware costs and enhances the efficiency of the database system.

5. Enhanced User Experience: Faster query response times and improved system performance contribute to a better user experience. Applications that deliver quick and responsive results are more likely to satisfy users and maintain their engagement.

6. SLA Compliance: Query optimization helps meet service level agreements (SLAs) by ensuring that the database system adheres to agreed-upon performance standards and response time requirements.

Implementation

To optimize queries, two indexing methods, namely B-tree indexing and hash indexing, have been implemented.B-tree indexing organizes data in a balanced tree structure, sorted in ascending or descending order. It efficiently handles duplicate keys by storing them in a sorted manner. This method is optimized for disk access, making it suitable for scenarios where data is stored on disk.On the other hand, hash indexing utilizes a hash function to compute the storage location of data directly. This method is ideal for exact match lookups and is primarily optimized for in-memory data access.

Using B-tree indexing

The selection of entries from the Customer table based on customer status and age has been optimized using indexing. Initially, without the index, Postgres performed a sequential scan, taking 4.863 ms to execute the query. After implementing a B-tree index named 'index_customer_id' on the 'customer_id_card_number' column, the execution time for the same query reduced to 4.275 ms, resulting in an optimization of 0.588 ms.

Triggers and functions

Triggers and functions serve as powerful tools in database systems, enabling the automation, validation, and customization of data processing tasks, thus enhancing data management and processing efficiency.

Triggers, associated with a table, automatically respond to specific events such as data insertions, updates, or deletions. They enforce constraints, validate data, audit changes, and initiate actions based on conditions. Triggers can execute before or after events, ensuring data integrity and consistency. Implemented using SQL statements, they allow for intricate logic to enforce business rules.

Functions, on the other hand, are named blocks of code used in databases to perform specific tasks, return values, or execute actions. They encapsulate reusable logic, accept input parameters, and can be written in various programming languages. Functions enhance code reuse, modularity, and maintainability by encapsulating complex logic into reusable units. They find application in queries, stored procedures, triggers, and other database objects, performing calculations, data transformations, or complex operations.

For instance, a trigger has been created for the checkout table. The trigger function 'calculate_fine()' is invoked after an insertion in the checkout table. This function updates the 'fine' column for the newly inserted row by calculating the fine amount, which is 5% of the book's price per day. The trigger and function collaboratively work to automatically calculate and assign a fine value for each new record in the checkout table. When triggered, the function determines the fine based on the actual due date and return date of the new record. This calculated fine is then stored as the specific record's fine value. This streamlined process ensures automatic generation and assignment of fines, eliminating the need for manual intervention.
